## Summary

This phase transforms the homepage from a lightweight landing page into a content-rich, keyword-optimized page that competes with established Lithuanian competitors like autosutartis.lt. The work spans four concerns: (1) new content sections (FAQ, stats, service description), (2) expanding existing copy with keyword depth, (3) fixing heading hierarchy for proper SEO structure, and (4) adding FAQPage JSON-LD structured data.

The existing codebase is well-structured for this work. Home.razor uses static SSR (no `@rendermode` directive), all user-visible text is stored in `Text.resx` resource strings, JSON-LD is already generated via Schema.NET 13.0.0 using the `Thing[]` + `ToHtmlEscapedString()` pattern, and the layout already has a footer in `MainLayout.razor`. The primary challenge is the volume of new Lithuanian content that must be keyword-informed and naturally written, plus a database query for the stats counter on a static SSR page.

**Critical finding:** Google restricted FAQ rich snippets to government and health websites in August 2023. FAQPage JSON-LD will NOT produce rich results for autodokas.lt. However, the structured data still provides value: it helps Google understand page content structure, benefits AI answer engines (Google AI Overviews, Bing Chat), and is low-effort to implement since Schema.NET already supports it. The FAQ section itself remains highly valuable for on-page SEO and user experience regardless of rich result eligibility.

**Primary recommendation:** Implement all content additions using existing patterns (Text.resx for strings, static SSR for rendering, Schema.NET for JSON-LD). Use `IDbContextFactory<AppDbContext>` to query contract count for the stats section. Focus most effort on writing high-quality Lithuanian copy informed by competitor keyword analysis.

### Pattern 3: Database Query for Stats Counter on Static SSR Page

**What:** Inject `IDbContextFactory<AppDbContext>` into the static SSR page and query the contract count in `OnInitializedAsync`.
**When to use:** For the stats/counters section showing real database data.

The existing Home.razor is static SSR (no `@rendermode` directive). It can still use dependency injection via `@inject`. Since the page renders once per request (no circuit/connection like InteractiveServer), a scoped `AppDbContext` via factory is appropriate.

**Example:**

```csharp
@inject IDbContextFactory<AppDbContext> DbFactory

@code {
    private int _contractCount;

    protected override async Task OnInitializedAsync()
    {
        // Query contract count
        using var context = await DbFactory.CreateDbContextAsync();
        _contractCount = await context.VehicleContracts
            .CountAsync(c => c.Status == VehicleContract.ContractStatus.Completed);

        // Existing JSON-LD generation...
    }
}
```

**Important:** Change `OnInitialized` to `OnInitializedAsync` since the DB query is async. The existing JSON-LD code can remain synchronous inside the same method.

### Pattern 6: Resource String Pattern for New Content

**What:** All new text goes in `Text.resx` with matching `Text.Designer.cs` auto-generation.
**When to use:** Every piece of new user-visible text.

The `.csproj` configures `PublicResXFileCodeGenerator` for Text.resx, so building the project (`dotnet build`) regenerates `Text.Designer.cs` automatically. New entries need:
1. Add `<data name="KeyName">` to `Text.resx`
2. Build project to regenerate `Text.Designer.cs`
3. Reference as `@Text.KeyName` in Razor

### Anti-Patterns to Avoid
- **Hardcoded Lithuanian text in Razor:** All text must go through `Text.resx` — the project has zero hardcoded strings in Home.razor
- **Accordion/collapsible FAQ:** User explicitly decided against this. No JS interaction for FAQ.
- **Keyword stuffing:** Google penalizes unnatural keyword repetition. Keywords must read naturally in Lithuanian.
- **Separate code-behind file for Home.razor:** The existing `@code {}` block pattern is used. Don't create `Home.razor.cs` unless the code grows unwieldy.

## Common Pitfalls

### Pitfall 1: FAQPage Rich Results Expectation Mismatch
**What goes wrong:** Team expects FAQ rich snippets in Google SERP after adding FAQPage JSON-LD.
**Why it happens:** Google restricted FAQ rich results to government and health websites in August 2023. autodokas.lt is not in either category.
**How to avoid:** Set correct expectations. FAQPage JSON-LD still adds value: (1) helps Google understand page structure, (2) benefits AI search engines (Google AI Overviews, Bing Copilot), (3) is low-effort since Schema.NET supports it, (4) competitor autosutartis.lt still uses it. Implement it, but don't expect SERP rich snippets.
**Warning signs:** If Rich Results Test shows "Not eligible for rich results" for FAQPage — this is expected, not an error.

### Pitfall 2: Text.Designer.cs Not Regenerated After Adding Resource Strings
**What goes wrong:** Build errors like "Text does not contain a definition for 'Faq1Question'" after adding entries to Text.resx.
**Why it happens:** `Text.Designer.cs` is auto-generated on build. If using an IDE that doesn't auto-build, or if the build is cached, the generated file may be stale.
**How to avoid:** Run `dotnet build` after modifying `Text.resx`. The `.csproj` has `PublicResXFileCodeGenerator` configured, so build always regenerates.
**Warning signs:** Compile errors referencing new resource string names.

### Pitfall 3: Static SSR Page with Async Database Query
**What goes wrong:** Using `OnInitialized` (synchronous) to run an async database query, causing a deadlock or requiring `.Result` anti-pattern.
**Why it happens:** The existing Home.razor uses synchronous `OnInitialized()` for JSON-LD generation. Adding a DB query requires async.
**How to avoid:** Change `OnInitialized()` to `OnInitializedAsync()` and use `await` for the database query. The existing synchronous code (JSON-LD construction) works fine inside an async method.
**Warning signs:** Using `.Result` or `.GetAwaiter().GetResult()` on async calls.

### Pitfall 4: MarkupString XSS Risk in FAQ Answers
**What goes wrong:** FAQ answers with internal links (e.g., `<a href="/contract">Pradėkite čia</a>`) require `MarkupString` to render HTML, which bypasses Blazor's HTML encoding.
**Why it happens:** Resource strings are plain text by default. HTML in resource strings needs explicit `MarkupString` cast.
**How to avoid:** Only use `MarkupString` for resource strings you control (Text.resx entries). Never use it for user-generated content. The same pattern is already safely used for `ContractReadOnlyBanner` in the project.
**Warning signs:** Links appearing as escaped HTML text (`&lt;a href=...`) in the rendered page.

### Pitfall 5: Heading Hierarchy Breaking Visual Design
**What goes wrong:** Changing `<h5>` to `<h3>` causes visual changes because browser default h3 styles differ from h5.
**Why it happens:** Although `.step-title` and `.feat-title` CSS classes define font size and weight, browser defaults for h3 include different margins than h5.
**How to avoid:** After the element swap, verify that `margin-top` and `margin-bottom` on `.step-title` and `.feat-title` are explicitly set in CSS. The current CSS sets `margin-bottom: 0.6rem` but doesn't set `margin-top`, which differs between h3 and h5 browser defaults.
**Warning signs:** Vertical spacing changes in step cards and feature cards.

### Pitfall 6: Footer Already Exists in MainLayout.razor
**What goes wrong:** Creating a duplicate footer when one already exists.
**Why it happens:** CONTEXT.md says "Add site-wide shared footer via layout component" but MainLayout.razor already has a `<footer class="footer-site">` with links to /terms-and-conditions and /privacy-policy plus copyright.
**How to avoid:** The existing footer already satisfies the requirement. Review the footer and only update the copyright text from "All rights reserved" to match the user's specification ('© 2024 AutoDokas.lt') if needed. Current footer uses `@DateTime.Now.Year` which auto-updates.
**Warning signs:** Two footers appearing on the page.

## Open Questions

1. **Schema.NET FAQPage MainEntity API**
   - What we know: `IFAQPage` interface exists in Schema.NET 13.0.0 (verified in DLL). `Question` and `Answer` types also exist.
   - What's unclear: The exact property signature for `MainEntity` on `FAQPage` — whether it accepts `List<IQuestion>` directly or requires `Values<>` wrapper.
   - Recommendation: The implementer should test compilation at implementation time. If `List<IQuestion>` doesn't work for MainEntity, try `new Values<ICreativeWork, IItemList>(questions.Cast<ICreativeWork>().ToList())` or set individual question entries. LOW risk — worst case, write FAQ JSON-LD as a raw JSON string if Schema.NET API is awkward.

2. **FAQ Answer Plain Text vs HTML**
   - What we know: FAQ answers need HTML links for display (`MarkupString` in Razor) but JSON-LD should contain plain text.
   - What's unclear: Whether to maintain two resource strings per answer (one with HTML, one plain) or strip HTML at runtime.
   - Recommendation: Use a single resource string with HTML and strip tags for JSON-LD using a simple regex or string replacement. This avoids maintaining duplicate strings.

3. **Stats Counter: Which Numbers to Show**
   - What we know: Competitor autosutartis.lt shows "2663 completed contracts" as a trust signal.
   - What's unclear: How many completed contracts AutoDokas has. If the number is very small, showing it could undermine trust.
   - Recommendation: Query `VehicleContracts.Count(c => c.Status == Completed)`. If the number is < 50, consider showing a different stat or omitting the section initially. Claude's discretion per CONTEXT.md.
